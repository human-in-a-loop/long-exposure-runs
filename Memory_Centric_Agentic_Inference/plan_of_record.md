# Plan of Record — Investigate and develop a memory-centric architecture for ag

**Created:** 2026-05-11T12:16:49Z

## Directive (verbatim)

Investigate and develop a memory-centric architecture for agentic LLM inference.

The central research question is whether future AI infrastructure should be designed around the movement, placement, reuse, compression, and lifetime management of model weights, KV caches, intermediate state, tool outputs, and agent trajectory state, rather than around arithmetic throughput alone. Generic LLM serving already faces pressure from memory capacity, memory bandwidth, data movement, interconnects, and power. Long-running agentic systems may make these constraints more important because they create persistent context, branch-and-merge execution, repeated access to prior work, heterogeneous tool use, verification loops, and durable multi-agent state.

Long-exposure has full authority to decide how to decompose and pursue the problem. It should choose subproblems, sources, models, simulations, experiments, artifacts, pivots, and stopping points internally. No user ratification, clarification, guidance, or unblock decisions will be provided during the live run. When ambiguity arises, make conservative assumptions, document them, and continue.

The run should aim to produce research-grade artifacts that would interest AI infrastructure, GPU/accelerator, compiler, operating-systems, distributed-systems, and datacenter developers. The final output should not be a generic survey. It should attempt to create defensible models, workload taxonomies, design principles, simulation artifacts, and architecture proposals, while clearly labeling what is sourced, what is derived, what is simulated, and what remains speculative.

Do not close off potential strategies prematurely, but keep the work centered on the memory-centric architecture problem. Explore and compare multiple plausible angles within that topic, including but not limited to:

- Agentic inference workload taxonomy: how agentic runs differ from chat, batch summarization, RAG, code agents, tool-using agents, and offline inference.
- Memory movement and lifetime models for weights, KV caches, prompt prefixes, tool outputs, retrieved context, branch state, and verification state.
- Scheduling abstractions: whether the right object is a request, job, kernel, model, cache page, context segment, or an agent trajectory with uncertain future branches.
- Heterogeneous memory hierarchy: HBM, GPU memory, CPU DRAM, CXL or pooled memory, NVMe tiers, remote memory, semantic caches, prefix caches, and durable workspace state.
- Runtime and compiler implications: whether agentic workflows need memory planning passes, context budget allocation, cache residency policies, branch-aware scheduling, and verification-aware state management.
- Energy and economics: when memory-centric placement and reuse policies reduce energy, latency, cost, or capacity pressure, and what workload or hardware trends would falsify the design.

Build inspectable artifacts in the workspace as the run progresses: literature maps, assumptions tables, cost models, simulators, workload generators, plots, policy comparisons, diagrams, audit notes, and falsification criteria. Keep topic-specific artifacts organized under a clear memory-centric project subdirectory when possible. Prefer simple, robust models and executable prototypes over brittle complexity.

Use Wolfram Engine or Mathematica through the configured Wolfram Language execution path for advanced symbolic mathematics, optimization, recurrence relations, queueing approximations, sensitivity analysis, or any computation where symbolic or high-precision tooling would help. The local `wolfram-bridge` tree under the workspace is a helper/skill and documentation interface for Wolfram Language work, not a replacement for Wolfram Language execution itself; use it when useful for Wolfram documentation lookup or bridge patterns. Use Python and other tools when they are better suited for simulation, plotting, data processing, or harness construction.

Success should look like a coherent research package: a clear problem formulation, a nontrivial taxonomy of agentic inference memory demands, at least one analytical model, at least one executable simulator or evaluation harness, comparative results for multiple policies or architectures, a concrete proposed architecture or set of architecture options, explicit failure modes, and a ranked agenda of research questions that could become papers, prototypes, or product experiments.

Operate autonomously for 100+ cycles unless stopped by the runtime. Make decisions internally. Do not wait for user approval.

Every milestone status, history, and judgment lives in `promise_ledger.jsonl`,
filtered by `milestone_id`. Run `promise_check` to materialize the current
state for the human; agents call it via Bash:

    python3 -m long_exposure.tools.promise_check .

The directive section above is **immutable** after creation. Goals and
milestones tables are mutable, but every edit must emit a ledger event with
`milestone_id: "_plan/<descriptive-change-name>"` so the audit trail is
complete.
