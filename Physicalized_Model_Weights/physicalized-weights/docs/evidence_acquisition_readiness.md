# Evidence-Acquisition Readiness

M-ACQUIRE-1 is a pre-collection screen for future shadow production dual-run or canary A/B dual-instrumented trace collection. It does not validate real evidence, does not score a measured margin, and cannot reopen the Phase 2 downgrade. A ready design means only that the collection plan can plausibly produce an admissible future evidence pack for the existing Phase 3 gate.

## Required Pre-Collection Controls

Before collection begins, the operator must lock the M-TRACE-1 schema version, select a known M-REOPEN-1 threshold scenario, and define a single policy-version window with update and rollback markers. The trace design must exclude raw prompts, raw content, user identifiers, tenant identifiers, API keys, email addresses, IP addresses, and other privacy-risk fields before any data is written. Privacy and provenance attestations must be generated by the collection system, not reconstructed after the fact.

The collection must use an M-INGEST-1 admissible path: shadow production dual-run or canary A/B dual-instrumented collection. It must capture the hybrid path and the programmable accelerator counterfactual over the same requests and policy window, with feature extraction, route decision, fallback, audit, update, rollback, health, drift, utilization, latency, and energy accounting. Hybrid and programmable accelerator energy must be measured; proxy, modeled, vendor-only, and local-only measurements can support diagnostics but cannot satisfy measured_terms.

The evidence-pack plan must generate a manifest, trace hash, schema version, source type, ingestion path, measurement status, threshold scenario id, provenance attestation, and privacy attestation. The package must be replayable by the M-EVIDENCEPACK-1 harness and then by the M-PIPELINE-1 gate. Hash mismatch, unknown threshold mapping, missing provenance, missing privacy attestation, or single-path logging are pre-collection blockers.

## Readiness Classes

- `ready_to_collect_candidate`: the design has the required structure for future admissible collection, but it is still not evidence.
- `repair_required_before_collection`: the design has an admissible structure but lacks repairable instrumentation, such as measured energy.
- `inadmissible_design`: the design would inevitably fail a fatal Phase 3 gate unless structurally redesigned.
- `diagnostic_only`: the design may be useful for debugging or calibration but cannot become reopen evidence because it is synthetic, vendor-only, proxy-only, or otherwise outside the eligible source path.

## Reopen Guard

Readiness is not evidence. Even a ready collection plan has no measured trace rows, no measured hybrid-minus-baseline margin, and no threshold crossing. The preserved future reopen condition remains:

`valid_package ∧ hash_match ∧ schema_compatible ∧ known_threshold_scenario ∧ valid_trace ∧ admissible_ingestion_path ∧ measured_terms ∧ production_or_shadow_or_canary_source ∧ provenance_attestation ∧ privacy_attestation ∧ threshold_crossed`

M-ACQUIRE-1 therefore preserves `actual_reopen_candidate_count = 0`.
